Architecture Blueprint

Overview

Below is a detailed, practical improvement guide for your Immich-Go GUI project, with a strong focus on making the project easier to maintain as immich-go evolves.

The main recommendation is:

Keep the UI in app.py for now, but extract a small, stable “backend/core” layer that knows how to translate GUI state into immich-go commands, environments, warnings, and execution plans.

This gives you decoupling without a large rewrite.

Section 01

1. Current Project Assessment

Your project already has several good foundations:

#01.1 What is already good

  1. GUI is fairly well structured
    - Sidebar navigation
    - Card-based forms
    - Simple/Advanced mode
    - Theme support
    - Dry-run preview
    - Command masking for secrets

  2. Secret handling is already better than many GUI wrappers
    - API key stored via OS keychain
    - Migration away from plain-text QSettings
    - Masking secrets in command preview
    - Passing secrets through environment variables at runtime

  3. Tests exist
    - Pure logic tests
    - GUI smoke tests
    - Command builder tests
    - Secret store tests

  4. Theme system is separated
    - theme.py is already a good example of decoupling
    - UI styling is not fully mixed into app logic

  5. The app is already thinking in terms of tabs and command domains
    - upload-folder
    - upload-gp
    - upload-immich
    - archive-folder
    - archive-immich
    - stack

That is a strong starting point.

Section 02

2. Main Risks and Weak Points

The biggest long-term risks are not UI-related. They are maintainability-related.

#02.1 2.1 immich-go CLI changes will break the GUI

Right now, the GUI directly knows CLI flag names:

python 
cmd_opts.append("--include-type=...")
cmd_opts.append("--manage-burst=...")
cmd_opts.append("--from-client-timeout=...")

This means:

  • If a flag is renamed, UI code breaks.
  • If a flag is removed, UI code may generate invalid commands.
  • If a flag default changes, UI behavior may silently become wrong.
  • If a new subcommand appears, UI support requires scattered changes.
  • If immich-go changes environment variable naming, runtime auth may break.

This is the biggest maintainability problem.

#02.2 2.2 UI knows too much about command construction

Currently app.py does all of these:

  • Builds widgets
  • Stores widget references
  • Reads widget state
  • Knows CLI flag names
  • Knows flag ordering
  • Knows which flags are global vs command-specific
  • Knows which flags are upload-only
  • Masks secrets
  • Builds environment variables
  • Launches processes
  • Checks running processes
  • Downloads binaries
  • Handles themes
  • Handles persistence

That is too many responsibilities for one UI file.

You do not need to split everything immediately, but you should separate the “what the user wants” from “how to talk to immich-go”.

#02.3 2.3 Process execution is fragile

Current execution opens an external terminal:

  • Windows: cmd /c start cmd /k ...
  • macOS: osascript
  • Linux: tries gnome-terminal, konsole, xfce4-terminal, xterm

Issues:

  1. Environment variables may not reliably propagate through all terminal launch methods.
  2. Process detection is based on scanning for immich-go in process names.
  3. The GUI cannot reliably show output, errors, progress, or cancellation.
  4. External terminal behavior differs across desktop environments.
  5. Dry-run output is not captured inside the GUI.

For a GUI wrapper, embedded execution is much more robust.

#02.4 2.4 Binary management needs hardening

Current binary management:

  • Downloads latest release from GitHub
  • Extracts binary
  • Checks version by running immich-go version

Missing:

  • Version pinning
  • Compatibility matrix
  • Checksum verification
  • Signature verification
  • Manual binary path selection
  • Fallback to PATH binary
  • Recovery from broken downloads
  • Clear compatibility warnings

Because immich-go changes often, “always latest” is risky.

#02.5 2.5 Configuration persistence is incomplete

The app currently saves only a few things:

  • Server URL
  • API key via keychain
  • Skip SSL
  • Theme

But many fields are not persisted:

  • Client timeout
  • Concurrent tasks
  • Device UUID
  • On errors
  • Pause jobs
  • Upload options
  • Archive options
  • Stack options
  • Advanced flags

Also, immich_go_gui_config.toml appears to be a legacy/reference config format that is not fully used by app.py.

This creates confusion:

  • Users may expect settings to persist.
  • The TOML file suggests a config model that the app does not fully implement.
  • Future import/export config features may become messy.
#02.6 2.6 Validation is too weak

Current validation mostly checks:

python 
if not re.match(r"^https?://.+", srv) or not api:
    return False

But the GUI should validate per active tab:

  • Upload folder path exists?
  • Google Takeout path is not empty?
  • Archive destination is selected?
  • Source Immich server/key provided for upload from-immich?
  • Date range format valid?
  • Extensions format valid?
  • Binary exists?
  • Binary is executable?
  • Server URL normalized?
  • API key not empty?

Right now, invalid commands may be generated and only fail at runtime.

#02.7 2.7 Secret handling can be improved further

You already mask secrets and use environment variables, which is good.

But build_command() still initially includes secrets in the command list:

python 
cmd_opts.append(f"--api-key={api}")
cmd_opts.append(f"--from-api-key={c['from-api-key'].text()}")

Then run_command() strips them.

A cleaner design is:

Never put secrets into the command argv in the first place.

The command builder should return:

python 
CommandPlan(
    argv=[...],          # no secrets
    env={...},           # secrets here
    display_argv=[...],  # safe preview
    warnings=[...]
)

This reduces the chance of accidental leakage in logs, previews, crash reports, or future features.

Section 05

5. Core Idea: UI State → Domain Request → CLI Command

The UI should not say:

python 
"--manage-burst=Stack"

The UI should say:

python 
state.upload_folder.manage_burst = "Stack"

Then the backend decides:

  • Which CLI version supports this flag
  • Whether the flag is called --manage-burst
  • Whether it should be emitted
  • Whether it was renamed
  • Whether it was removed
  • Whether it needs a warning
#05.1 5.1 Introduce a CommandPlan

Create a simple object representing the final execution plan.

python 
from dataclasses import dataclass, field


@dataclass
class CommandPlan:
    argv: list[str]
    env: dict[str, str] = field(default_factory=dict)
    display_argv: list[str] = field(default_factory=list)
    warnings: list[str] = field(default_factory=list)
    errors: list[str] = field(default_factory=list)
    tab_key: str = ""
    dry_run: bool = False

Example:

python 
plan = CommandPlan(
    argv=[
        "upload",
        "from-folder",
        "--server=http://localhost:2283",
        "--manage-burst=Stack",
        "/photos",
    ],
    env={
        "IMMICH_GO_UPLOAD_API_KEY": "secret",
    },
    display_argv=[
        "upload",
        "from-folder",
        "--server=http://localhost:2283",
        "--manage-burst=Stack",
        "/photos",
    ],
    warnings=[],
    errors=[],
    tab_key="upload-folder",
    dry_run=False,
)

The UI only asks:

python 
plan = backend.build_plan_from_gui_state(self.collect_gui_state(), dry_run=False)

Then:

  • Preview uses plan.display_argv
  • Runner uses plan.argv
  • Runner uses plan.env
  • UI shows plan.warnings
  • UI blocks execution if plan.errors exists
Section 06

6. Decouple immich-go Version Changes

This is the most important long-term improvement.

#06.1 6.1 Problem

immich-go may:

  • Rename flags
  • Remove flags
  • Change defaults
  • Add new commands
  • Change environment variable names
  • Change version output
  • Change help output
  • Change release asset names

Your GUI should not break every time.

#06.2 6.2 Solution: Versioned CLI adapters

Create a compatibility layer.

Concept

python 
class ImmichGoAdapter:
    def build_upload_folder(self, request, config) -> CommandPlan:
        raise NotImplementedError

    def build_upload_google_photos(self, request, config) -> CommandPlan:
        raise NotImplementedError

    def build_upload_immich(self, request, config) -> CommandPlan:
        raise NotImplementedError

    def build_archive_folder(self, request, config) -> CommandPlan:
        raise NotImplementedError

    def build_archive_immich(self, request, config) -> CommandPlan:
        raise NotImplementedError

    def build_stack(self, request, config) -> CommandPlan:
        raise NotImplementedError

Then create version-specific adapters:

python 
class ImmichGoAdapter_v0_31(ImmichGoAdapter):
    ...


class ImmichGoAdapter_v0_32(ImmichGoAdapter_v0_31):
    ...


class ImmichGoAdapter_v0_33(ImmichGoAdapter_v0_32):
    ...

The GUI does not care which adapter is used.

#06.3 6.3 Adapter selection

When the app starts:

  1. Find binary.
  2. Run immich-go version.
  3. Parse version.
  4. Choose adapter.

Example:

python 
def get_adapter(version: str) -> ImmichGoAdapter:
    if version.startswith("0.31"):
        return ImmichGoAdapter_v0_31()
    if version.startswith("0.32"):
        return ImmichGoAdapter_v0_32()
    if version.startswith("0.33"):
        return ImmichGoAdapter_v0_33()

    # Unknown newer version:
    # Use latest known adapter, but warn.
    return ImmichGoAdapter_v0_33()

You can also add a warning:

text 
immich-go version 0.40 detected.
This GUI was tested up to version 0.33.
Some options may behave differently.

This is much better than silently breaking.

7. Introduce CLI Capability Profiles

#07 7. Introduce CLI Capability Profiles

Version adapters are good. Capability profiles are even better.

Instead of only saying:

python 
if version == "0.32":
    ...

You say:

python 
if profile.supports("upload.from-folder.manage-burst"):
    ...

Example capability profile:

python 
@dataclass
class CliCapabilities:
    version: str
    supports_pause_immich_jobs: bool = True
    supports_on_errors_numeric: bool = True
    supports_from_client_timeout: bool = True
    supports_api_trace: bool = True
    supports_device_uuid: bool = True
    supports_upload_from_icloud: bool = False
    supports_upload_from_picasa: bool = False
    supports_archive_from_icloud: bool = False
    supports_archive_from_picasa: bool = False
    supports_stack_epson_fastfoto: bool = True

Then command builder checks capabilities:

python 
if capabilities.supports_from_client_timeout and request.from_client_timeout != 20:
    argv.append(f"--from-client-timeout={request.from_client_timeout}m")

If a future version removes the flag, you update the capability profile, not the UI.

Section 12

12. Introduce a Flag Specification Layer

#12 12. Introduce a Flag Specification Layer

This is the strongest long-term defense against CLI changes.

Instead of writing many if statements:

python 
if c["manage-burst"].currentText() != "NoStack":
    cmd_opts.append(f"--manage-burst={c['manage-burst'].currentText()}")

Define flag rules declaratively.

Example:

python 
FLAG_SPECS = [
    {
        "ui_key": "manage_burst",
        "flag": "--manage-burst",
        "type": "choice",
        "default": "NoStack",

#14.3 Runner design

Create immich_go_runner.py:

python 
from PySide6.QtCore import QObject, QProcess, Signal


class ImmichGoRunner(QObject):
    output_received = Signal(str)
    error_received = Signal(str)
    started = Signal()
    finished = Signal(int, str)
    failed = Signal(str)

    def __init__(self, binary_path: str, parent=None):
        super().__init__(parent)
        self.binary_path = binary_path
        self.process = None

    def run_plan(self, plan):
        if self.process is not None:
            self.failed.emit("Another immich-go process is already running.")
            return

        self.process = QProcess(self)
        self.process.setProcessEnvironment(plan.env)

        self.process.readyReadStandardOutput.connect(self._read_stdout)
        self.process.readyReadStandardError.connect(self._read_stderr)
        self.process.started.connect(self.started.emit)
        self.process.finished.connect(self._on_finished)
        self.process.errorOccurred.connect(self._on_error)

        self.process.start(self.binary_path, plan.argv)

    def cancel(self):
        if self.process is not None:
            self.process.kill()

    def _read_stdout(self):
        if self.process is None:
            return
        data = self.process.readAllStandardOutput().data().decode("utf-8", errors="replace")
        self.output_received.emit(data)

    def _read_stderr(self):
        if self.process is None:
            return
        data = self.process.readAllStandardError().data().decode("utf-8", errors="replace")
        self.error_received.emit(data)

    def _on_finished(self, exit_code, exit_status):
        self.finished.emit(exit_code, str(exit_status))
        self.process = None

    def _on_error(self, error):
        self.failed.emit(str(error))
        self.process = None

            self._add_upload_folder_args(argv, state.get("upload-folder", {}))

            self._add_upload_connection_args(argv, env, config, warnings)
            self._add_upload_google_photos_args(argv, state.get("upload-gp", {}))

        elif tab_key == "upload-immich":
            argv.extend(["upload", "from-immich"])
            self._add_upload_connection_args(argv, env, config, warnings)
            self._add_upload_immich_args(argv, env, state.get("upload-immich", {}))

        elif tab_key == "archive-folder":
            argv.extend(["archive", "from-folder"])
            self._add_archive_folder_args(argv, state.get("archive-folder", {}))

        elif tab_key == "archive-immich":
            argv.extend(["archive", "from-immich"])
            self._add_archive_connection_args(argv, env, config, warnings)
            self._add_archive_immich_args(argv, state.get("archive-immich", {}))

        elif tab_key == "stack":
            argv.append("stack")
            self._add_stack_connection_args(argv, env, config, warnings)
            self._add_stack_args(argv, state.get("stack", {}))

        else:
            errors.append("No executable tab selected.")

        if dry_run and "--dry-run" not in argv:
            argv.append("--dry-run")

        display_argv = self.mask_display_command([self.binary_path] + argv)

        return CommandPlan(
            argv=argv,
            env=env,
            display_argv=display_argv,
            warnings=warnings,
            errors=errors,
            tab_key=tab_key,
            dry_run=dry_run,
        )

    def validate_state(self, state: dict) -> list[str]:
        errors = []
        tab_key = state.get("active_tab", "")
        config = state.get("config", {})

        if tab_key == "config":
            return errors

        server = config.get("server_url", "").strip()
        api_key = config.get("api_key", "").strip()

        if tab_key != "archive-folder":
            if not server:
                errors.append("Server URL is required.")
            if not api_key:
                errors.append("API key is required.")

        if tab_key == "upload-folder":
            if not state.get("upload-folder", {}).get("path", "").strip():
                errors.append("Source folder or ZIP path is required.")

        if tab_key == "upload-gp":
            if not state.get("upload-gp", {}).get("path", "").strip():
                errors.append("Google Takeout source is required.")

        if tab_key == "archive-folder":
            folder = state.get("archive-folder", {})
            if not folder.get("path", "").strip():
                errors.append("Source path is required.")
            if not folder.get("write_to", "").strip():
                errors.append("Destination folder is required.")

        if tab_key == "archive-immich":
            if not state.get("archive-immich", {}).get("write_to", "").strip():
                errors.append("Destination folder is required.")

        return errors

    def _add_upload_connection_args(self, argv, env, config, warnings):
        server = config.get("server_url", "").strip()
        api_key = config.get("api_key", "").strip()

        if server:
            argv.append(f"--server={server}")
            env["IMMICH_GO_UPLOAD_SERVER"] = server

        if api_key:
            # Do not put API key into argv.
            env["IMMICH_GO_UPLOAD_API_KEY"] = api_key

        if config.get("skip_ssl"):
            argv.append("--skip-verify-ssl")
            warnings.append(
                "SSL verification is disabled. "
                "Use only on trusted networks or self-hosted test servers."
            )

    def mask_display_command(self, argv: list[str]) -> list[str]:
        masked = []
        secret_flags = {
            "--api-key",
            "--from-api-key",
            "--admin-api-key",
        }

        skip_next = False
        for part in argv:
            if skip_next:
                skip_next = False
                continue

            if part in secret_flags:
                masked.append(part)
                masked.append("********")
                skip_next = True
                continue

            hidden = False
            for flag in secret_flags:
                if part.startswith(f"{flag}="):
                    masked.append(f"{flag}=********")
                    hidden = True
                    break

            if not hidden:
                masked.append(part)

        return masked

This is intentionally simplified, but it shows the correct separation.

Section 40

40. Final Recommendation

#40 40. Final Recommendation

The single most important improvement is this:

Create a small backend layer that converts GUI state into a version-aware CommandPlan, and make the UI responsible only for collecting state and displaying results.

That one change solves most of your long-term concerns:

  • Easier maintenance
  • Safer secret handling
  • Better testing
  • Easier adaptation to immich-go breaking changes
  • Better process execution
  • Cleaner UI code
  • Easier future feature development

For now, do not over-split the project.

Use this minimal structure:

text 
app.py                  # UI
theme.py                # Theme
immich_go_backend.py    # Command/version/validation logic
immich_go_runner.py     # Process/binary execution logic

Then gradually introduce:

  1. CommandPlan
  2. Backend state models
  3. Version adapters
  4. CLI capability profiles
  5. Embedded QProcess runner
  6. Binary pinning
  7. Full settings persistence
  8. Compatibility checker script

That gives you a practical path from the current codebase to a much more maintainable project without stopping active development.
