# GitHub Actions & Releases Documentation

This document serves as a personal reference for testing, running, and managing GitHub Actions workflows and Releases.

## 1. Triggering Workflows Manually (`workflow_dispatch`)

When testing workflows on a separate branch (before merging to `main`), the easiest method is to manually trigger the workflow via the GitHub UI. This requires the `workflow_dispatch` trigger in your YAML file.

### Adding the Trigger
Ensure your workflow YAML (e.g., `.github/workflows/release.yml`) has this at the top:
```yaml
on:
  workflow_dispatch:  # <--- This adds the manual trigger button
  push:
    tags:
      - 'v*'
```

### Running the Workflow
1. Push your branch to GitHub (`git push -u origin your-branch-name`).
2. Go to the **Actions** tab on your GitHub repository.
3. On the left sidebar, click the name of your workflow (e.g., "Build and Release Apps").
4. On the right, click the **Run workflow** dropdown.
5. Select your test branch from the dropdown list.
6. Click the green **Run workflow** button.

## 2. Triggering via Tags (Automated Releases)

The `push: tags: - 'v*'` block means the workflow automatically runs whenever a Git tag starting with "v" (like `v1.0.0`) is pushed to GitHub. This is typically used for actual production releases.

### How to Create and Push a Tag
If you want to simulate a release or make an actual release:
1. Make sure all your code is committed.
2. Create the tag locally: 
   ```bash
   git tag v1.0.0
   ```
3. Push the tag to GitHub: 
   ```bash
   git push origin v1.0.0
   ```
4. GitHub Actions will detect the new tag, run the workflow, and (if configured) attach built binaries to a new Release page.

### Testing with a Beta Tag
You can safely test release pipelines on a non-main branch by tagging a commit with a beta tag:
```bash
git tag v0.0.1-beta
git push origin v0.0.1-beta
```
This triggers the workflow. Once testing is complete, you can delete the release from the GitHub UI and delete the tag locally/remotely if desired.

## 3. Dealing with Artifacts vs Releases

* **Artifacts:** Temporary files (usually kept for 90 days) generated by individual jobs in a workflow (like the Windows `.exe` build). You can download them directly from the Actions run summary page.
* **Releases:** Permanent snapshots of your software. The `softprops/action-gh-release` step in the workflow takes the artifacts and automatically publishes them to a formal Release page (e.g., `github.com/user/repo/releases`).

## 4. Building Locally with Nuitka and UV

If you want to build the executable locally on your own machine without triggering GitHub Actions, you can use `uv` and `nuitka`. 

First, ensure you have your dependencies installed and nuitka added to your dev environment:
```bash
uv sync
uv add --dev nuitka
```

Then, depending on your operating system, run the appropriate build command:

**For Linux (Standalone):**
```bash
uv run python -m nuitka --assume-yes-for-downloads --standalone --enable-plugin=pyside6 --include-data-files=immich-go-gui.png=immich-go-gui.png app.py
```

**For Windows (Standalone, No Console - Official Build):**
First, ensure you have `pillow` to convert the PNG to ICO:
```bash
uv add --dev pillow
uv run python -c "from PIL import Image; Image.open('immich-go-gui.png').save('immich-go-gui.ico')"
```
Then run the full Nuitka build with metadata (this matches the GitHub Actions CI):
```bash
uv run python -m nuitka --assume-yes-for-downloads --standalone --windows-console-mode=disable --enable-plugin=pyside6 --windows-icon-from-ico=immich-go-gui.ico --company-name="Shitan198u" --product-name="Immich-Go GUI" --file-version="1.0.0.0" --product-version="1.0.0.0" --file-description="Immich-Go Graphical User Interface" --copyright="MIT License" --include-data-files=immich-go-gui.png=immich-go-gui.png app.py
```

**For Windows (Debugging - Console Attached):**
If you want to build the executable locally to test crashes or view logs, use `--windows-console-mode=attach` so PowerShell displays the stack trace:
```bash
uv run python -m nuitka --assume-yes-for-downloads --standalone --windows-console-mode=attach --enable-plugin=pyside6 --windows-icon-from-ico=immich-go-gui.ico --include-data-files=immich-go-gui.png=immich-go-gui.png app.py
```

**For macOS (App Bundle):**
```bash
uv run python -m nuitka --assume-yes-for-downloads --macos-create-app-bundle --enable-plugin=pyside6 --include-data-files=immich-go-gui.png=immich-go-gui.png app.py
```
# Developer Best Practices for Immich-Go GUI

With the new automated workflows (CI, Release Drafter, CodeQL, Pre-commit, Dependabot) now active on your repository, your development lifecycle has leveled up! 

To get the most out of these tools and maintain a pristine codebase, you should adopt the following best practices:

## 1. Stop Committing Directly to `master`
Now that you have branch protection and CI checks, you should treat `master` as your "production-ready" code.
* **Do this instead**: When building a feature or fixing a bug, create a new branch locally:
  ```bash
  git checkout -b feature/dark-mode
  # Make changes, commit, and push
  git push origin feature/dark-mode
  ```
* Open a **Pull Request (PR)** on GitHub. This allows your CI (Pytest) and CodeQL to run against your changes in an isolated environment before they touch `master`.

## 2. Use Labels on Your Pull Requests!
Since we installed **Release Drafter**, it relies entirely on the labels you assign to your Pull Requests.
* Before merging a PR, always assign it a label on the right side of the GitHub UI.
* If you label a PR as `enhancement`, it goes under "🚀 Features" in your release notes.
* If you label a PR as `bug`, it goes under "🐛 Bug Fixes".
* If you forget to add a label, Release Drafter won't know where to put it in the changelog!

## 3. Setup Your Local Pre-commit Hooks
We added a `.pre-commit-config.yaml` file, but you need to activate it locally.
* Run this command once on your computer:
  ```bash
  uv run pre-commit install
  ```
* **What it does**: Every time you type `git commit`, it will quickly scan your code. It will use `ruff` to automatically format your Python files, fix trailing whitespace, and stop you from accidentally committing massive files (like large uncompressed images). If it fixes something, you just `git add` and commit again.

## 4. Let GitHub Handle the Releases
Previously, you might have created Git tags manually (`git tag v1.0.0 && git push origin v1.0.0`). 
* **Do this instead**: Go to your GitHub repository's **Releases** page. You will see a "Draft" release sitting at the top.
* Click the pencil icon to edit the draft.
* Type in your new version number (e.g., `v1.2.0`) in the tag box.
* Click **Publish release**. 
* This instantly creates the tag for you *and* triggers your `release.yml` to compile the Windows/Mac/Linux binaries and attach them.

## 5. Reviewing Dependabot PRs
Once a week, **Dependabot** will check if packages like `pyside6` or GitHub Actions (like `checkout@v4`) have new versions.
* It will open an automatic Pull Request.
* **Don't just merge blindly!** Wait a couple of minutes for your CI tests (`test_app.py`) to finish running on the PR. 
* If the tests pass (Green Checkmark ✅), you can safely hit merge knowing the update didn't break your GUI!

---

> [!TIP]
> By following these 5 steps, your `master` branch will never break, your code will always be beautifully formatted, and your release notes will write themselves!

# Commonly Used Git Commands

Here is a quick reference for the most common Git commands you will use during development:

## Branching & Merging
* `git checkout -b <branch-name>`: Create and switch to a new branch (e.g., `git checkout -b feat/new-button`).
* `git switch <branch-name>` or `git checkout <branch-name>`: Switch to an existing branch.
* `git branch`: List all local branches.
* `git merge <branch-name>`: Merge the specified branch into your current active branch.

## Saving Changes
* `git status`: Check the state of your working directory (shows modified and untracked files).
* `git add .`: Stage all current changes for the next commit.
* `git commit -m "feat: your message"`: Commit your staged changes with a descriptive message.

## Syncing with Remote (GitHub)
* `git pull origin <branch-name>`: Fetch and download content from the remote repository and immediately update the local repository to match that content.
* `git push origin <branch-name>`: Upload your local branch commits to the remote repository.
* `git fetch --prune`: Downloads objects and refs from the remote, and removes any remote-tracking references that no longer exist on the remote.

# Release Please: Things to Keep in Mind

Since we have migrated to **Release Please**, our release workflow is now entirely driven by our commit history. Keep the following rules in mind:

## 1. Use Conventional Commits
Release Please reads your commit messages (or PR titles) to determine the next version number. You **must** use these prefixes:
* `feat: ...` -> Triggers a **Minor** release bump (e.g., `1.1.0` -> `1.2.0`). Use for new features.
* `fix: ...` -> Triggers a **Patch** release bump (e.g., `1.1.0` -> `1.1.1`). Use for bug fixes.
* `chore: ...` -> No version bump, but can still appear in the changelog depending on configuration. Use for maintenance.
* `feat!: ...` or `fix!: ...` (Notice the `!`) -> Triggers a **Major** release bump (e.g., `1.1.0` -> `2.0.0`). Use for breaking changes.

## 2. The "Release PR" is Automatic
When you push or merge a `feat` or `fix` to the `master` branch, Release Please will automatically open (or update) a Pull Request named `chore: release vX.Y.Z`. 
* **Do not edit this PR manually** unless you know what you are doing. Release Please maintains it.
* It will automatically update `pyproject.toml` and the `CHANGELOG.md` file.

## 3. Merging Creates the Release
When you are ready to publish a new version of the app to users:
1. Go to the open "Release PR" created by Release Please.
2. Click **Merge pull request**.
3. Release Please will automatically create the Git tag (e.g., `v1.2.0`) and publish the GitHub Release page.
4. Your `release.yml` GitHub Action will detect the new tag and automatically build and upload the executables for Windows, Linux, and macOS.

You can convert Markdown files (such as `review.md`) into standalone, interactive HTML files featuring nested collapsible sections, a real-time search filter, a sticky Table of Contents sidebar, dark/light theme switching, and copy buttons on code blocks.
